# MASTER_ARCHITECTURE_PLAN_V1

## 0. Статус документа
- Версия: `v1.0`
- Дата: `2026-03-05`
- Тип: `Normative Architecture Plan`
- Область действия: оркестратор, локальные CLI-агенты, память/RAG, транспорт, ресурсная безопасность

## 1. Мандат и инженерная рамка

### 1.1 Цель
Установить детерминированную архитектуру исполнения задач в многоагентной системе, исключив передачу исполняемого кода от оркестратора агентам и минимизировав стохастическое поведение LLM в критическом контуре.

### 1.2 Анти-цель
Система не должна опираться на метафорические или недетерминированные конструкции в runtime-контуре (`"душа"`, `"жар"`, `"энергетический тупик"` как рабочие сущности запрещены).

### 1.3 Нормативные термины
- `MUST`: обязательное требование.
- `MUST NOT`: строго запрещено.
- `SHOULD`: рекомендуется, отклонение требует обоснования.
- `MAY`: допустимо как опция.

## 2. Ключевые архитектурные решения

1. Оркестратор генерирует только декларативные `Task Spec`-контракты.
2. Локальный агент исполняет только внутреннюю предкомпилированную логику DevKit.
3. Передача shell/python/sql-кода от оркестратора агенту запрещена.
4. Патчи применяются только при успешном `SHA-256 integrity pinning`.
5. Поиск контекста реализуется как `Hybrid RAG` (`Dense + Sparse + RRF`).
6. Агенты работают как эфемерные процессы через `systemd socket activation`.
7. Межпроцессное взаимодействие: `ZeroMQ + MessagePack`.
8. Против перегрузки: `Credit-Based Flow Control (CBFC)`.

## 2.1 Корректировка к исходному черновику (обязательное соответствие)
Этот `MASTER` трактуется как план уровня `L0` и обязан сохранять структуру исходного каркаса:
1. `P0`: детерминированная инженерия + работа в физических лимитах машины.
2. Гибридное ядро памяти через эволюцию `Archivator_Agent`.
3. Федеративная локальная векторная топология с эфемерной активацией.
4. Динамическая оркестрация микропакетов с аппаратным backpressure.
5. Трехуровневая интеграция ядра (`L0/L1/L2`).

Отклонение локальных реализаций от этих пяти опор `MUST NOT` происходить без явного ADR.

## 3. Task Spec Layer

### 3.1 Инварианты
- Оркестратор `MUST` задавать только желаемое состояние, а не процедуру.
- Task Spec `MUST` быть сериализуемым в `YAML` или `MessagePack`.
- Цель `goal` `MUST` быть однострочной и без управляющих ветвлений.
- Спецификация `MUST NOT` содержать runtime-код проверок.

### 3.2 Минимальная схема Task Spec
```yaml
spec_version: 1
task_id: TSK-2026-000001
goal: "Синхронизировать индекс памяти агента с указанным commit"
constraints:
  derivation_only: true
  code_injection_forbidden: true
preconditions:
  - type: file_exists
    path: "./devkit/apply_patch.sh"
  - type: min_version
    component: "git"
    version: ">=2.39"
artifacts:
  patch_path: "patches/tsk-2026-000001.patch"
  patch_sha256: "<64-hex>"
  target_ref: "refs/heads/main"
limits:
  timeout_ms: 30000
  max_output_tokens: 2048
expected_result:
  status: success
  changed_files_max: 20
```

### 3.3 Пассивные предусловия
- `preconditions` описывают статические условия.
- Агент `MUST` самостоятельно выполнять проверки безопасными локальными примитивами.
- При несоответствии предусловий агент `MUST` завершаться по `fail-fast`.
- Авто-установка зависимостей в момент выполнения `MUST NOT` выполняться.

### 3.4 Derivation-Only
- Агенту разрешены только преобразования входных данных в выходные согласно контракту.
- Добавление новой бизнес-логики во время исполнения `MUST NOT` происходить.

### 3.5 Integrity Pinning
- Каждый артефакт патча `MUST` иметь поле `sha256`.
- Перед применением патча агент `MUST` сверить хэш байт-в-байт.
- При несовпадении хэша выполнение `MUST` быть остановлено.

### 3.6 Применение патчей
- Базовый механизм: `git apply --index --3way`.
- При непримиримом конфликте агент `MUST` вернуть `conflict_detected` и не продолжать.

## 4. Память и Hybrid RAG

### 4.0 Гибридный Архиватор (исходная опора)
- `Archivator_Agent` является базовым узлом гибридной памяти и `MUST` вести двойной контур:
  1. физические неизменяемые архивы (`zip/tgz`) для долговременного хранения;
  2. семантический локальный индекс для retrieval-контуров.
- Логика ZIP-архивирования `MUST` сохраняться как отдельный обязательный путь, не заменяемый векторным индексом.

### 4.1 Двухконтурная память
- Локальные CLI-агенты: `SQLite + sqlite-vec + FTS5`.
- Центральный оркестратор: `LanceDB` для больших архивов.

### 4.2 Причины выбора
- `sqlite-vec` для агента: автономность, один бинарный файл, транзакционность, низкий overhead.
- `LanceDB` для оркестратора: масштаб дисковых коллекций и быстрый columnar I/O.

### 4.3 Логика гибридного поиска
1. Dense retrieval: векторный KNN (`cosine` или `L2`).
2. Sparse retrieval: `FTS5/BM25` по точным терминам.
3. Слияние: `Reciprocal Rank Fusion`.

Формула RRF:
`score(d) = Σ_i 1 / (k + rank_i(d))`, где `k = 60` по умолчанию.

### 4.4 Политика индексации
- Документы памяти (`LAM_DATA_Src`, `LAM_MEM.CORE`, runbooks, task logs) `MUST` проходить embedding pipeline.
- Индекс `SHOULD` обновляться инкрементально при каждом подтвержденном task completion.

## 5. Эфемерные агенты и термодинамика

### 5.1 Runtime-модель
- Always-on демоны для агентного контура `MUST NOT` использоваться.
- Запуск агента `MUST` происходить только по факту входящего запроса (on-demand).
- Для индивидуальных агентных узлов тяжелые постоянно активные векторные демоны `MUST NOT` использоваться.

### 5.2 Systemd Socket Activation
Поток выполнения:
1. `*.socket` слушает порт/UDS при нулевом потреблении агентом RAM.
2. Входящее соединение активирует `*.service`.
3. Агент получает дескриптор сокета, обрабатывает Task Spec.
4. Агент завершает процесс сразу после ответа.

### 5.3 Cold start контроль
- Виртуальное окружение и зависимости `SHOULD` быть предсозданы (`bootstrap.sh`, `.venv`).
- 95-й перцентиль cold start `SHOULD` быть < `500 ms`.

## 6. Транспорт и сериализация

### 6.1 Протокол
- Межагентный транспорт `MUST` быть `ZeroMQ`.
- Базовая топология: `ROUTER (оркестратор) <-> DEALER (агент)`.
- Центральный брокер сообщений `MUST NOT` быть обязательным элементом.

### 6.2 Формат данных
- Внутренний payload `MUST` использовать `MessagePack`.
- JSON для межпроцессного hot-path `MUST NOT` использоваться.

### 6.3 Минимальный пакет
```text
frame[0]: routing_id
frame[1]: msg_type (READY|TASK|ACK|ERROR|CREDIT)
frame[2]: msgpack_payload
```

## 7. Защита от стохастического паралича: CBFC

### 7.1 Правила
1. Агент при старте отправляет `CREDIT(n)`.
2. Оркестратор декрементирует кредит при отправке каждого `TASK`.
3. Если кредит = 0, отправка на этот узел запрещена.
4. После завершения задачи агент отправляет `ACK + CREDIT(+1)`.
5. Неактивные/зависшие узлы исключаются по таймауту heartbeat.

### 7.2 Эффект
Локальный сбой одного агента изолируется; каскадное переполнение очередей предотвращается.

## 8. Оркестратор: router.py и P0-безопасность

### 8.0 Трехуровневая интеграция ядра
- `L0` (LAM / Shadow BIOS): базовые правила маршрутизации и системные ограничения.
- `L1` (Captain Bridge): агрегирует только отфильтрованные high-level выжимки и принимает финальные решения.
- `L2` (Peripheral Agents): изолированные исполнительные капсулы с локальной памятью и контрактным исполнением.

Границы между `L0/L1/L2` `MUST` быть явными и проверяемыми тестами/контрактами.

### 8.1 Router Policy
- Роутинг `MUST` быть деревом явных правил (`if/else`), а не скрытой промпт-эвристикой.
- Приоритет `P0`:
  1. Техническая безопасность (ресурсы, время, целостность).
  2. Биологическая безопасность оператора.
  3. Выполнение функциональной миссии.

### 8.2 Circuit breakers
- Все LLM-вызовы `MUST` иметь:
  - `timeout_ms`
  - `max_tokens`
  - лимит ретраев
- При превышении лимитов задача `MUST` переводиться в fallback-путь.

### 8.3 Биомонитор как аппаратная обратная связь
- Сигналы Copilot Interface интерпретируются как инженерные telemetry-события.
- При критическом пороге истощения оркестратор `MUST`:
  1. остановить выдачу новых Task Spec,
  2. обнулить выдачу кредитов перегруженным контурам,
  3. перевести агентов в idle,
  4. потребовать явную ручную авторизацию на продолжение.

## 9. SLO, метрики, аудит

### 9.1 SLO (начальные)
- Успешная валидация Task Spec: `>= 99.5%`.
- Ошибки integrity mismatch: `0` допустимых bypass.
- p95 task completion (без LLM): `< 2 s`.
- p95 socket-activation startup: `< 500 ms`.

### 9.2 Аудит
- Каждая задача `MUST` логироваться как цепочка:
  `task_id -> spec_hash -> artifact_hash -> apply_result -> commit_ref`.
- Логи `SHOULD` быть структурированными (`msgpack/jsonl`) и подписанными timestamp + node_id.

## 10. План внедрения

1. Фаза R (Research Gate, обязательная перед фиксацией локальных стеков):
   - benchmark транспорта: `ZeroMQ` vs `gRPC` vs `FastAPI` для эфемерных сценариев;
   - benchmark локального векторного движка: `FAISS` vs `LanceDB` vs `SQLite-vec/SQLite-VSS`;
   - проектирование и проверка `Wake-on-Demand` триггеров ОС.
2. Фаза A (Task Spec Core): схема, валидатор, fail-fast, hash pinning.
3. Фаза B (Patch Runtime): `git apply --3way`, конфликтные статусы, rollback policy.
4. Фаза C (Memory): гибрид `архив + семантический индекс` с per-agent изоляцией.
5. Фаза D (Transport): асинхронный микропакетный контур и binary envelope.
6. Фаза E (Flow Control): CBFC, heartbeat, outlier isolation.
7. Фаза F (P0 Safety): circuit breakers, биомонитор, hard-stop policy.

## 10.1 Реализация через существующий скелет
- Реализация `MUST` идти через существующие узлы экосистемы и их текущие контракты.
- Создание новых агентов `MUST NOT` выполняться до прохождения `Anti-Sprawl Gate`.
- Локальные планы интеграции `MUST` фиксировать owner, reuse-точки и Task Spec пакет.
- Операционный реестр делегации вынесен в:
  `LOCAL_INTEGRATION_DELEGATION_PLAN_V1.md`.

## 11. Критерии приемки
- Нет передачи исполняемого кода от оркестратора к агентам.
- Любой патч без валидного `sha256` отвергается.
- Агент стартует только через socket activation.
- Hybrid RAG возвращает комбинированный top-k по RRF.
- При кредитах `0` оркестратор не отправляет задачи узлу.
- При P0-сигнале истощения активируется hard-stop без обходов.

## 12. Риски и контрмеры
- Риск: ложные срабатывания fail-fast.
  - Контрмера: точные коды ошибок + dry-run валидатор.
- Риск: деградация индекса из-за плохих embeddings.
  - Контрмера: периодический eval-recall@k и re-embedding pipeline.
- Риск: рост cold start latency.
  - Контрмера: slim dependencies, prebuilt wheels, warm cache.

## 13. Итого
Эта версия плана фиксирует переход к проверяемой архитектуре: декларативный Task Spec, криптографическая целостность, гибридная память, эфемерное исполнение, бинарный транспорт и строгий контроль потока. Концептуальные метафоры допускаются только в документации верхнего уровня и не участвуют в runtime-решениях.

## 14. Статус соответствия исходному черновику
- Гибрид `Archivator_Agent` (`ZIP + semantic index`) зафиксирован как обязательный.
- Федеративная изоляция памяти по агентам зафиксирована.
- Эфемерные серверы/Wake-on-Demand зафиксированы.
- Трехуровневая модель `L0/L1/L2` зафиксирована.
- Research Phase как обязательный gate перед локальной технологической фиксацией добавлен.
